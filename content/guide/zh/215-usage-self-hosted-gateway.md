# 2.5 自建 AI 大模型中转站

> 本文由 **卡网大全 [cardnav.xyz](https://cardnav.xyz)** 提供。

自建中转站通常适合有技术基础的人。你自己购买官方额度或准备可用账号，再用反代理、中转程序或内部中转站把模型能力分发给团队、朋友或自己的多个工具使用。

## 核心优势

- **额度可集中管理**：可以把官方额度统一分配给多个工具、项目或成员。
- **成本可控**：团队内部可以按需限制额度、记录调用量和控制预算。
- **接入灵活**：可以对接内部应用、代码助手、自动化脚本和多个客户端。
- **可做权限控制**：比直接把原始账号或主 Key 分给别人更容易管理。

## 局限与风险

- **需要运维能力**：服务器、域名、HTTPS、日志、限速、密钥管理都需要自己维护。
- **风控压力更集中**：多个用户或账号共用同一个中转入口，可能触发官方防滥用策略。
- **账号责任更复杂**：一旦团队成员滥用额度，可能影响主账号或上游额度。
- **安全边界要清楚**：中转服务本身会经过所有请求，必须保护好日志和密钥。

## 适合人群

- 有基础 Linux、Docker、反向代理和 API 运维经验的人
- 小团队想统一管理模型额度和 Key 的场景
- 想把模型能力接入内部工具、自动化流程或多个客户端的人

## 是否值得自建

### 值得做
- 团队需要统一分发 Key、限制额度和查看调用量
- 内部工具需要稳定接入多个模型或多个客户端

### 不建议一开始就做
- 只是个人轻量使用，优先考虑官方账号或成熟中转站
- 没有服务器、HTTPS、日志、安全和异常处理经验时，维护成本会很快超过省下的钱

### 关键边界
- 不要直接共享主账号或主 Key
- 尽量给成员分发独立 Key，并保留用量和异常记录

## 推荐项目

## Sub2API
<!-- badge="订阅转换" icon="api" -->

把订阅或账号凭证转换成 OpenAI 兼容接口，适合已经有可用上游账号、希望快速接入 API 客户端的用户。

### 适合场景
- 想把已有账号或订阅额度接入 Chatbox、Cherry Studio、代码插件等工具
- 需要先用较轻量方式验证自建中转流程

### 注意事项
- 部署前先确认上游账号和订阅来源合规、稳定
- 不要把主账号、主 Key 或敏感日志暴露给团队成员

[查看项目](https://github.com/Wei-Shaw/sub2api)

## CliProxyAPI
<!-- badge="CLI 代理" icon="vps" -->

面向 CLI 和 Agent 使用场景的代理接口项目，适合把模型能力分发给命令行工具、开发工作流和内部自动化任务。

### 适合场景
- 代码助手、命令行工具和 Agent 工作流需要统一接入模型
- 团队希望集中管理访问密钥、额度和调用入口

### 注意事项
- 需要配合 HTTPS、访问密钥、限速和日志策略一起部署
- 长期使用前先用真实开发任务测试稳定性、兼容性和额度消耗

[查看项目](https://github.com/router-for-me/CLIProxyAPI)

## 实操步骤

1. 先选购海外云服务器，具体主机选择可以参考 [云服务器推荐](./322-tool-vps.md)。
2. 准备官方账号、可用额度或上游 API Key，并确认其使用条款和风控边界。
3. 从上方推荐项目里选择更适合你使用场景的中转程序。
4. 配置 HTTPS、访问密钥、额度限制、日志保留策略和异常告警。
5. 给团队成员分发独立 Key，不要直接共享主账号或主 Key。
6. 定期检查调用量、失败率、异常请求和上游账号状态。

## 使用建议

自建中转站不是“更便宜的万能方案”，它更像一个需要持续维护的内部基础设施。只有当你确实需要团队共享、统一接入和权限控制时，它才值得做；个人轻量使用通常没必要一开始就走这条路。
