# CardNav 向导 Markdown 渲染说明

本文说明 `content/guide/` 下的向导 Markdown 如何渲染，方便后续新增或维护 Guide 文章。

## 文件位置

向导文章统一放在（其中 `<LANG>` 代表语言的缩写，例如 `zh`）：

```text
content/guide/<LANG>/
```

文件名建议使用 `数字-英文-slug.md`，例如：

```text
200-choose-usage-method.md
211-usage-api-relay.md
```

渲染时会用文件名前缀数字排序，并把文件名去掉数字后的部分作为页面 slug。

## Frontmatter

每篇文章顶部使用 YAML frontmatter：

```markdown
---
title: "2.1 AI 大模型 API 中转站"
description: "通过 API 中转站快速获得大模型调用能力，适合低门槛试用、多模型接入和轻量开发调试。"
parent:
  slug: choose-usage-method
next:
  slug: usage-ready-account
---
```

字段含义：

```ts
type GuideFrontmatter = {
  title: string; // 页面标题，优先写完整语义，适合 SEO 和目录展示
  description: string; // 页面摘要，用于列表、SEO 描述和分享摘要
  parent?: { slug: string }; // 上级文章，标题自动从目标文章读取
  next?: { slug: string }; // 下一篇文章，标题自动从目标文章读取
};
```

`title` 和 `description` 统一使用双引号，避免英文、俄文等内容里的 `: ` 触发 YAML 解析问题。`title` 应该是完整语义标题。章节页保留编号，例如 `2.1 AI 大模型 API 中转站`，不要只写 `API 中转站`。`parent` 和 `next` 只写 `slug`，标题由渲染逻辑自动读取目标文章的 `title`。

## 正文结构

正文优先使用标准 Markdown，先保证原文在 GitHub 里也能自然阅读，再由 Guide 网页渲染做增强。页面主标题用一个 H1：

```markdown
# 2.1 AI 大模型 API 中转站
```

普通章节使用 `##`、`###` 和列表即可。正文里的相对 Markdown 链接会自动改成站内 Guide 链接：

```markdown
[选择 AI 订阅使用方式](./200-choose-usage-method.md)
```

渲染后会指向对应的 `/guide/...` 页面。

用户可见正文只写对读者有判断价值的信息，不解释页面结构或交互方式。不要写“下面用几张卡片做总览”“点击卡片查看详细说明”“本节会渲染为卡片”这类元叙述。入口页导语应直接帮助用户分流，例如：

```markdown
先按使用门槛和风险边界判断路线：只想快速试用，优先看中转站或成品账号；已有账号但卡在付款，重点看第三方代充；准备长期重度使用，再考虑官方订阅充值或自建中转站。
```

## 卡片结构

连续的 `##` 章节可以渲染成卡片。卡片标题来自 `##` 标题，卡片内容来自该标题后面的正文。

只要 `##` 标题下面第一段非空内容是属性注释，这个章节就会变成卡片：

```markdown
## 2.1 AI 大模型 API 中转站
<!-- badge="上手最快" icon="api" -->

托管账号注册与付款，直接获取 API Key 接入第三方客户端使用。

### 核心优势
- 免海外手机号验证，无需折腾复杂代理
- 按量计费，适合先小额试用

### 主要风险
- 对话内容会经过中转服务商
- 需要防范假冒高配模型、服务掺水和服务商跑路

### 适合人群
- 调试代码、轻度非涉密对话、想快速体验多模型 API 的个人用户和开发者

[点击了解详情](./211-usage-api-relay.md)
```

属性注释支持这些字段：

```ts
type GuideCardAttrs = {
  badge?: string; // 卡片顶部标签
  icon?: string; // 卡片右上角图标名
  imageAspect?: string; // 卡片图片区域比例，例如 3.44/1
};
```

卡片跳转链接写在该 `##` 段落最后一行，使用单独一行标准 Markdown 链接。链接文案按真实动作填写，例如 `[点击了解详情](...)`、`[查看指南](...)`、`[前往注册](...)`、`[查看项目](...)`。网页渲染会把这条链接提取为整张卡片的点击目标，并在卡片底部显示同文案的行动提示；GitHub 等普通 Markdown 渲染器则会正常显示这条链接。

有跳转链接的卡片会显示行动提示并具备可点击样式；没有跳转链接的卡片只作为信息卡展示。不要在同一张卡片最后放多个跳转链接。

`[点击了解详情](./211-usage-api-relay.md)` 会自动转换为对应 Guide 页面链接。外部链接会保留原 URL，并自动加点击统计属性。

链接打开规则：

- 站内链接本页打开，包括 `/guide/...`、`/shops`、`#anchor` 和 `https://cardnav.xyz/...`
- 完整 URL 且不是 `cardnav.xyz` 的链接新页面打开，并自动添加 `target="_blank"` 和 `rel="noopener noreferrer"`
- 卡片段落最后一行跳转链接和正文 Markdown 链接遵守同一套规则

`icon` 只支持内置图标名，未知值不会渲染。当前可用值包括：

```text
api
bank
card
cart
cash
check
gift
globe
key
model
phone
route
shield-alert
support
target
toolbox
vpn
vps
```

卡片图片使用标准 Markdown 图片语法，不写进属性注释。这样同一份 Markdown 在 GitHub 里也能正常显示图片：

```markdown
## AnyRouter 公益站点
<!-- badge="公益额度" imageAspect="3.44/1" -->

![AnyRouter 赞助商标志](../../../public/anyrouter.png)

支持 GPT、Claude 等模型，每日登录赠送免费额度，适合先低成本试用 API 中转。

[前往注册](https://anyrouter.top/register?aff=JA5h)
```

网页渲染时会把卡片正文开头的第一张独立图片提取到卡片图片区，并把 `../../../public/xxx.png` 转成线上可访问的 `/xxx.png`。GitHub 渲染时则保留标准 Markdown 图片。

`imageAspect` 只控制网页卡片里的图片区比例，使用 `宽/高` 格式，例如 `3.44/1`。图片缩放方式由渲染样式统一决定：位图固定 `cover` 裁剪填充，SVG 固定 `contain` 完整展示。

图片写法规则：

- **必须用标准 Markdown 图片**：写 `![图片说明](../../../public/example.png)`，不要写 `image="..."` 或 `imageAlt="..."` 注释属性
- **路径优先兼容 GitHub**：Guide 文件位于 `content/guide/zh/`，引用 `public/` 下资源时用 `../../../public/xxx`
- **图片放在卡片正文开头**：属性注释后第一张独立 Markdown 图片会被网页提取为卡片图片；如果图片不在开头，就按普通正文图片渲染
- **比例放在注释属性**：网页卡片图片区域比例写 `imageAspect="3.44/1"`；没有图片时不要写这个属性
- **缩放方式不在 Markdown 里配置**：位图固定裁剪填充，SVG 固定完整展示，文章不要为单张图片单独定义缩放方式
- **alt 写真实对象**：图片说明写品牌、项目或截图对象，例如 `![RightCode 赞助商标志](../../../public/rightcode.webp)`，不要写“图片”“logo”

推荐入口、赞助商、工具项目等卡片可以放图片；普通选择卡片不强制放图，避免页面噪声过多。

## 卡片写法建议

卡片标题应直接写完整主题，尤其是承载 SEO 意图的 Guide 入口页：

```markdown
## 2.3 AI 大模型自备账号第三方代充
```

卡片正文保持短而可扫读，优先使用 `###` 做真实分组标题，例如：

- `### 核心优势`：这条路径为什么值得选
- `### 主要风险`：下单或使用前最该注意什么
- `### 适合人群`：谁最适合这条路径
- `### 注意事项`：推荐项目、工具或商家入口需要提前确认什么

`###` 在 GitHub 中是普通小标题，在网页卡片里会渲染成分组标题和分隔线。参考 `100-choose-model.md`：

```markdown
## 顶级海外大模型
<!-- badge="能力上限" icon="model" -->

如果你的目标是追求行业最强的通用和编程能力，优先推荐 GPT、Claude、Gemini 等海外主流大模型。

### 1. 使用门槛
- **网络与验证**：需要稳定的海外 IP 和真实的手机号进行账号注册。
- **支付风控**：官网订阅常遇拒付，需配合海外应用商店礼品卡完成支付。

### 2. 适合谁
- **更适合**：寻求最前沿的编程开发、通用逻辑推理及成熟生态体验的用户
- **不适合**：预算有限，且不希望花时间折腾海外网络与付款卡的用户
```

不要在卡片正文里重复写一遍完整标题；标题本身已经会进入卡片标题区域。

不要把跳转提示混进说明性正文；需要卡片跳转时，只在该 `##` 段落最后一行保留标准 Markdown 链接。

## 内容写法边界

Guide 文章面向新手用户，正文应该帮助用户判断“我该选哪条路、要准备什么、风险在哪里”。写作时遵守这些边界：

- **先给判断**：入口页先按门槛、风险、使用周期或预算帮用户分流，不解释排版
- **术语要解释**：遇到“日抛号”“成品号”“共享号”“团队席位号”等市场叫法，就近解释含义、来源和风险
- **风险要具体**：不要只写“有风险”，说明可能来自账号来源、付款异常、多人共用、多地登录、售后短或平台风控
- **不要承诺稳定**：第三方账号、代充、中转和自建项目都只写适用场景和复核建议，不写保证可用
- **用户语言优先**：保留 API、Key、KYC、IP 等关键词时，就近说明它对注册、付款、登录或封号有什么影响

## 属性注释不会显示

Markdown 渲染器关闭了原始 HTML 输出，普通 HTML 不会直接渲染。卡片属性注释只由 Guide 渲染逻辑读取，不会作为正文显示。

如果注释格式写错，例如缺少引号，系统会把该章节当普通 Markdown 渲染。
